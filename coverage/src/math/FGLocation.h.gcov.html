      66             : /** FGLocation holds an arbitrary location in the Earth centered Earth fixed
      67             :     reference frame (ECEF). The coordinate frame ECEF has its center in the
      68             :     middle of the earth. The X-axis points from the center of the Earth towards
      69             :     a location with zero latitude and longitude on the Earth surface. The Y-axis
      70             :     points from the center of the Earth towards a location with zero latitude
      71             :     and 90 deg East longitude on the Earth surface. The Z-axis points from the
      72             :     Earth center to the geographic north pole.
      73             : 
      74             :     This class provides access functions to set and get the location as either
      75             :     the simple X, Y and Z values in ft or longitude/latitude and the radial
      76             :     distance of the location from the Earth center.
      77             : 
      78             :     It is common to associate a parent frame with a location. This frame is
      79             :     usually called the local horizontal frame or simply the local frame. It is
      80             :     also called the NED frame (North, East, Down), as well as the Navigation
      81             :     frame. This frame has its X/Y plane parallel to the surface of the Earth.
      82             :     The X-axis points towards north, the Y-axis points east and the Z-axis
      83             :     is normal to the reference spheroid (WGS84 for Earth).
      84             : 
      85             :     Since the local frame is determined by the location (and NOT by the
      86             :     orientation of the vehicle IN any frame), this class also provides the
      87             :     rotation matrices required to transform from the Earth centered (ECEF) frame
      88             :     to the local horizontal frame and back. This class "owns" the
      89             :     transformations that go from the ECEF frame to and from the local frame.
      90             :     Again, this is because the ECEF, and local frames do not involve the actual
      91             :     orientation of the vehicle - only the location on the Earth surface. There
      92             :     are conversion functions for conversion of position vectors given in the one
      93             :     frame to positions in the other frame.
      94             : 
      95             :     The Earth centered reference frame is NOT an inertial frame since it rotates
      96             :     with the Earth.
      97             : 
      98             :     The cartesian coordinates (X,Y,Z) in the Earth centered frame are the master
      99             :     values. All other values are computed from these master values and are
     100             :     cached as long as the location is changed by access through a non-const
     101             :     member function. Values are cached to improve performance. It is best
     102             :     practice to work with a natural set of master values. Other parameters that
     103             :     are derived from these master values are calculated only when needed, and IF
     104             :     they are needed and calculated, then they are cached (stored and remembered)
     105             :     so they do not need to be re-calculated until the master values they are
     106             :     derived from are themselves changed (and become stale).
     107             : 
     108             :     Accuracy and round off
     109             : 
     110             :     Given,
     111             : 
     112             :     - that we model a vehicle near the Earth
     113             :     - that the Earth surface average radius is about 2*10^7, ft
     114             :     - that we use double values for the representation of the location
     115             : 
     116             :     we have an accuracy of about
     117             : 
     118             :     1e-16*2e7ft/1 = 2e-9 ft
     119             : 
     120             :     left. This should be sufficient for our needs. Note that this is the same
     121             :     relative accuracy we would have when we compute directly with
     122             :     lon/lat/radius. For the radius value this is clear. For the lon/lat pair
     123             :     this is easy to see. Take for example KSFO located at about 37.61 deg north
     124             :     122.35 deg west, which corresponds to 0.65642 rad north and 2.13541 rad
     125             :     west. Both values are of magnitude of about 1. But 1 ft corresponds to about
     126             :     1/(2e7*2*pi) = 7.9577e-09 rad. So the left accuracy with this representation
     127             :     is also about 1*1e-16/7.9577e-09 = 1.2566e-08 which is of the same magnitude
     128             :     as the representation chosen here.
     129             : 
     130             :     The advantage of this representation is that it is a linear space without
     131             :     singularities. The singularities are the north and south pole and most
     132             :     notably the non-steady jump at -pi to pi. It is harder to track this jump
     133             :     correctly especially when we need to work with error norms and derivatives
     134             :     of the equations of motion within the time-stepping code. Also, the rate of
     135             :     change is of the same magnitude for all components in this representation
     136             :     which is an advantage for numerical stability in implicit time-stepping.
     137             : 
     138             :     Note: Both GEOCENTRIC and GEODETIC latitudes can be used. In order to get
     139             :     best matching relative to a map, geodetic latitude must be used.
     140             : 
     141             :     @see Stevens and Lewis, "Aircraft Control and Simulation", Second edition
     142             :     @see W. C. Durham "Aircraft Dynamics & Control", section 2.2
     143             : 
     144             :     @author Mathias Froehlich
     145             :   */
